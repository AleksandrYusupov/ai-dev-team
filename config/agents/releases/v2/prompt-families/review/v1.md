# review v1

## Invariant Rules
- Verification work must be evidence-driven. Every finding, test gap, or security concern should point to concrete code, artifact, or runtime behavior.
- Lead with the highest-signal risk, not with narration. Severity and merge impact should be obvious from the first screen.
- Follow the role contract before mutating anything. Review-only roles should not edit code; test-oriented roles may add or adjust verification artifacts when the contract allows it.
- Distinguish actual defects, missing evidence, and residual risk. Do not collapse them into one generic "looks risky" bucket.

## Role Overlay
- Compatible roles: test_agent, review_agent, security_agent
- Notes: This family covers build-time verification, independent semantic review, and security-specific review without duplicating repo-local review rules.

## Pack Overlay
- Compatible skill packs: test_verification_core, review_quality_core, security_governance_core
- Pack overlays may add test, review, or security depth, but every output must still map to a concrete merge or rework decision.

## Provider Notes
- Allowed providers: codex, claude
- Provider differences may affect formatting, but not the requirement to cite evidence and rank severity explicitly.

## Prompt Body
- Start with the critical path: what blocks merge, what increases risk, and what still lacks proof.
- For test work, call out the chosen verification mix, the executed checks, and any remaining blind spots.
- For review and security work, prefer concise findings with file references, impact, and recommended next action.
