## 1. Общая характеристика проекта

### Что это такое
Система AI Dev Team — это **orchestration control plane** для автоматизации инженерного рабочего процесса с использованием AI-агентов (Claude Code, Codex). Система получает задачи из Linear, проводит их через жизненный цикл (triage → spec → build → review → human decision → merge → deploy), используя Temporal как durable workflow engine и Postgres как источник execution truth.

### Масштаб кодовой базы
| Метрика | Значение |
|---------|----------|
| TypeScript файлов (без dist/node_modules) | 114 |
| Общий LOC (TypeScript) | ~40 400 |
| Тестовых файлов | 36 |
| Config/Migration LOC (YAML + SQL) | ~5 000 |
| Миграций БД | 12 |
| Пакетов в monorepo | 5 (shared, config, db, control-api, workflow-worker, runner-host) |

### Стек технологий
- **Runtime:** Node.js 22+, TypeScript 5.8
- **API:** Fastify 5
- **Workflow Engine:** Temporal (TypeScript SDK)
- **БД:** PostgreSQL (Kysely query builder)
- **Monorepo:** pnpm workspaces
- **CI:** GitHub Actions

## 4. Архитектурный анализ

### Сильные стороны архитектуры

1. **Правильное разделение truth sources.** Linear → intent, Temporal+Postgres → execution, GitHub → code. Это фундаментально правильное решение, которое предотвращает drift между «что показывается человеку» и «что реально происходит».

2. **Config version pinning.** Каждый issue run использует замороженную версию правил. Новые правила не ломают in-flight работу. Это серьёзная production-grade практика.

3. **Outbox pattern.** Все side effects (Linear comments, lease creation, etc.) идут через `workflow_effect_outbox`, что обеспечивает at-least-once delivery и idempotency.

4. **Immutable context packs.** Context packs кешируются по fingerprint'у и не мутируются. Это гарантирует, что build и review видят одну и ту же картину мира.

5. **Transition validator.** Каждый status transition проходит через `validateTransition()` — единственный легальный gate. Это предотвращает invalid state transitions.

6. **Continue-as-new в workflow.** `IssueLifecycleWorkflow` корректно использует Temporal's `continueAsNew` после 50 команд, что предотвращает переполнение execution history.

7. **Timing-safe auth.** Bearer token comparison использует `timingSafeEqual`, что правильно для security-sensitive кода.

8. **Input validation.** Fastify JSON schema validation на всех runner-host routes — malformed payloads отклоняются до бизнес-логики.

### Потенциальные архитектурные слабости

1. **Singleton DB connection в activities.** `getDb()` в `activities/index.ts` использует module-level singleton без graceful shutdown. При hot-reload или worker restart это может привести к connection leaks.

2. **Нет circuit breaker для внешних вызовов.** `postLinearComment` делает fetch без timeout, retry policy или circuit breaker. При Linear downtime outbox executor будет ретраить бесконечно.

3. **Нет rate limiting на webhook endpoints.** `/webhooks/linear` и `/webhooks/github` не имеют rate limiting. Burst of webhook deliveries может перегрузить inbox.

4. **Монолитный `index.ts` для shared.** 40+ литеральных union типов и DTOs в одном файле. Пока не критично, но при росте будет сложно навигировать.

## 6. Работоспособность по компонентам

### ✅ Полностью работоспособные компоненты

- **Webhook ingress pipeline**: Linear/GitHub → raw body capture → HMAC verify → delivery dedup → `raw_event_inbox` → inbox processor → normalized events → lifecycle commands
- **Workflow config system**: YAML manifests → validation → publish transaction → version pinning → runtime reads
- **Transition validation engine**: trigger matching → guard evaluation → artifact policy → status transition → audit write → outbox emission → projection update
- **Temporal workflows**: `IssueLifecycleWorkflow` (long-lived per issue), `CommentResponseWorkflow` (short-lived per comment), `IntegrationOnboardingWorkflow` (skeleton)
- **Knowledge sync**: Obsidian vault → note snapshots в Postgres с wiki-link resolution
- **Context pack assembly**: Issue contract + comments + note excerpts + repo guidance → deterministic fingerprinted bundle
- **Repository registry**: CRUD + project mappings + inspection API
- **Runner host infrastructure**: Process lifecycle, worktree management, MCP pool, artifact staging
- **Provider adapters**: Codex и Claude CLI adapters с process tree management, cancel support
- **Runner protocol**: Full lifecycle (manifest → claim → start → heartbeat → complete/fail → cancel-ack)
- **Observability foundation**: `agent_execution_metadata`, `agent_metrics_daily`, system-health, stuck-issues, lifecycle snapshot/journey views

### ⚠️ Частично работоспособные компоненты

- **Outbox executor**: `create_runner_lease` и `release_runner_lease` — полностью; `post_linear_comment` — реализован, но работает только для review decision summaries
- **Build → Review bridge**: Build completion корректно переводит в `agent_review` через outbox, но review completion → `needs_human_decision` runtime path не замкнут
- **Integration auth plane**: DB schema полностью, metadata reads — да, live OAuth / secret operations — нет

### ❌ Не реализованные компоненты (из исходной постановки)

- **Полноценный state-sync обратно в Linear** (статусы, PR links, agent activities, project updates)
- **PR/branch/check синхронизация** (Phase 8+)
- **Deploy/release automation** (Phase 8+)
- **Repo provisioning** (Phase 9+)
- **Multi-repo choreography** (Phase 9+)
